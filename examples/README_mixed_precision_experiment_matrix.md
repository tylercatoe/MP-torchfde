# Mixed-Precision Fractional Experiment Matrix

This README explains `examples/mixed_precision_experiment_matrix.py`, a reproducible harness for comparing mixed-precision adjoint training modes in `torchfde`.

## What This Script Is

`mixed_precision_experiment_matrix.py` is a controlled benchmark + quality check for fractional adjoint training.

It runs multiple precision/scaler configurations under the same seed, model shape, data, and optimizer settings, then reports:

- final train loss
- final validation loss
- validation delta vs fp32 baseline
- NaN/Inf event count
- mean epoch time
- training throughput (samples/sec)
- peak GPU memory

## What Data It Uses

This is **not MNIST/STL10/CIFAR**.  
It uses a **synthetic teacher-student regression setup**:

- Inputs are random vectors sampled from `torch.randn(...)`.
- Targets are generated by a frozen **teacher fractional model** that uses `fdeint` (direct solver).
- The benchmarked student uses `fdeint_adjoint` (adjoint solver) with different precision/scaler policies.

Why this setup:

- isolates solver precision behavior from dataset preprocessing noise
- makes runs fast and reproducible
- provides a stable fp32 reference for quality comparison

## Configurations Tested

The script automatically runs:

- `fp32_unscaled` (`loss_scaler=False`)
- `bf16_unscaled` (`loss_scaler=False`) if `torch.cuda.is_bf16_supported()`
- `fp16_safe` (`loss_scaler=False`)
- `fp16_dynamic` (`loss_scaler=DynamicScaler(torch.float16)`)

If CUDA is unavailable, only `fp32_unscaled` runs.

## Quick Start

From repo root:

```bash
python examples/mixed_precision_experiment_matrix.py --device cuda
```

Smaller smoke run:

```bash
python examples/mixed_precision_experiment_matrix.py \
  --device cuda \
  --epochs 2 \
  --train-samples 1024 \
  --val-samples 256 \
  --batch-size 128
```

Save results:

```bash
python examples/mixed_precision_experiment_matrix.py \
  --device cuda \
  --epochs 6 \
  --csv-out mp_matrix.csv \
  --json-out mp_matrix.json
```

## One-Command Multi-Sweep Driver

To run all suggested follow-up sweeps (scale, beta, discretization, seed) and get one consolidated summary:

```bash
python examples/run_mixed_precision_sweeps.py --device cuda
```

This creates a timestamped output folder under `examples/sweep_outputs/` containing:

- per-run logs and raw outputs
- `consolidated_results.csv/json` (one row per config per run)
- `best_by_run.csv/json` (recommended config for each run)

Quick preview without running:

```bash
python examples/run_mixed_precision_sweeps.py --device cuda --dry-run
```

Run only selected sections:

```bash
python examples/run_mixed_precision_sweeps.py --device cuda --sections beta seed
```

## Command-Line Arguments

Core run controls:

- `--device {auto,cuda,cpu}`: device selection (`auto` chooses CUDA if available)
- `--seed`: global seed (default `2026`)
- `--epochs`: training epochs per configuration

Dataset size:

- `--train-samples`: number of synthetic training examples
- `--val-samples`: number of synthetic validation examples
- `--batch-size`: train/val batch size
- `--targets-batch-size`: batch size used when generating teacher targets

Model and solver:

- `--dim`: input/state dimension
- `--hidden`: hidden width for vector field MLP
- `--out-dim`: regression output dimension
- `--beta`: fractional order
- `--t-final`: terminal integration time
- `--step-size`: integration step size
- `--adjoint-method`: student method (default `predictor-f`)
- `--teacher-method`: teacher method (default `predictor`)

Optimizer:

- `--lr`: learning rate
- `--weight-decay`: AdamW weight decay

Output:

- `--csv-out <path>`: optional CSV summary
- `--json-out <path>`: optional JSON summary

## Interpreting Output

The script prints one summary table with one row per configuration.

Recommended decision rule:

- First require `nan_inf_events == 0` and `status == ok`.
- Then compare `final_val_loss` and `val_vs_fp32_pct` against `fp32_unscaled`.
- Among acceptable-quality configs, choose lowest `mean_epoch_s` and/or lowest `peak_mem_mib`.

Practical guidance:

- `fp16_dynamic` is often best if `fp16_safe` is unstable.
- `bf16_unscaled` is often the easiest stable speedup if BF16 is supported.
- `fp32_unscaled` is your quality baseline.

## Notes and Caveats

- This harness measures relative behavior in a controlled synthetic setting.
- Final production choice should be confirmed on your real task/dataset.
- TF32 is disabled for cleaner precision comparisons.
- The script keeps running even if one config errors; failing rows are marked in `status`.
